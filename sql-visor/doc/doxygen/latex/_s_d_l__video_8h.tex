{\ttfamily S\+D\+L\+\_\+\+W\+I\+N\+D\+O\+W\+\_\+\+S\+H\+O\+WN} is ignored by \doxyref{S\+D\+L\+\_\+\+Create\+Window()}{p.}{_s_d_l__video_8h_a42565088bdb739a5353fd67abbe235d2}. The S\+D\+L\+\_\+\+Window is implicitly shown if S\+D\+L\+\_\+\+W\+I\+N\+D\+O\+W\+\_\+\+H\+I\+D\+D\+EN is not set. {\ttfamily S\+D\+L\+\_\+\+W\+I\+N\+D\+O\+W\+\_\+\+S\+H\+O\+WN} may be queried later using \doxyref{S\+D\+L\+\_\+\+Get\+Window\+Flags()}{p.}{_s_d_l__video_8h_a4445733f04a1a9dd0e063cd4738f285f}.

On Apple\textquotesingle{}s mac\+OS, you {\bfseries{must}} set the N\+S\+High\+Resolution\+Capable Info.\+plist property to Y\+ES, otherwise you will not receive a High-\/\+D\+PI Open\+GL canvas.

If the window is created with the {\ttfamily S\+D\+L\+\_\+\+W\+I\+N\+D\+O\+W\+\_\+\+A\+L\+L\+O\+W\+\_\+\+H\+I\+G\+H\+D\+PI} flag, its size in pixels may differ from its size in screen coordinates on platforms with high-\/\+D\+PI support (e.\+g. i\+OS and mac\+OS). Use \doxyref{S\+D\+L\+\_\+\+Get\+Window\+Size()}{p.}{_s_d_l__video_8h_a5cbfffcfec91c22a7ca95fd1d5163db5} to query the client area\textquotesingle{}s size in screen coordinates, and \doxyref{S\+D\+L\+\_\+\+G\+L\+\_\+\+Get\+Drawable\+Size()}{p.}{_s_d_l__video_8h_ac21851bbd91760c5bab92594a58edba3} or \doxyref{S\+D\+L\+\_\+\+Get\+Renderer\+Output\+Size()}{p.}{_s_d_l__render_8h_abfc0c9a50d9d1870ab7d271b7a73d8ab} to query the drawable size in pixels.

If the window is set fullscreen, the width and height parameters {\ttfamily w} and {\ttfamily h} will not be used. However, invalid size parameters (e.\+g. too large) may still fail. \doxyref{Window}{p.}{struct_window} size is actually limited to 16384 x 16384 for all platforms at window creation.

If the window is created with any of the S\+D\+L\+\_\+\+W\+I\+N\+D\+O\+W\+\_\+\+O\+P\+E\+N\+GL or S\+D\+L\+\_\+\+W\+I\+N\+D\+O\+W\+\_\+\+V\+U\+L\+K\+AN flags, then the corresponding Load\+Library function (S\+D\+L\+\_\+\+G\+L\+\_\+\+Load\+Library or S\+D\+L\+\_\+\+Vulkan\+\_\+\+Load\+Library) is called and the corresponding Unload\+Library function is called by \doxyref{S\+D\+L\+\_\+\+Destroy\+Window()}{p.}{_s_d_l__video_8h_a72609295a4faee0587268339ccff80c2}.

If S\+D\+L\+\_\+\+W\+I\+N\+D\+O\+W\+\_\+\+V\+U\+L\+K\+AN is specified and there isn\textquotesingle{}t a working Vulkan driver, \doxyref{S\+D\+L\+\_\+\+Create\+Window()}{p.}{_s_d_l__video_8h_a42565088bdb739a5353fd67abbe235d2} will fail because \doxyref{S\+D\+L\+\_\+\+Vulkan\+\_\+\+Load\+Library()}{p.}{_s_d_l__vulkan_8h_a9fcc0942c8eaa67c3eff2e3526304b5b} will fail.

If S\+D\+L\+\_\+\+W\+I\+N\+D\+O\+W\+\_\+\+M\+E\+T\+AL is specified on an OS that does not support Metal, \doxyref{S\+D\+L\+\_\+\+Create\+Window()}{p.}{_s_d_l__video_8h_a42565088bdb739a5353fd67abbe235d2} will fail.

On non-\/\+Apple devices, S\+DL requires you to either not link to the Vulkan loader or link to a dynamic library version. This limitation may be removed in a future version of S\+DL.

{\footnotesize\ttfamily D\+E\+C\+L\+S\+P\+EC void$\ast$ S\+D\+L\+C\+A\+LL S\+D\+L\+\_\+\+G\+L\+\_\+\+Get\+Proc\+Address (\begin{DoxyParamCaption}\item[{const char $\ast$}]{proc }\end{DoxyParamCaption})}

Get an Open\+GL function by name.

If the GL library is loaded at runtime with \doxyref{S\+D\+L\+\_\+\+G\+L\+\_\+\+Load\+Library()}{p.}{_s_d_l__video_8h_a398df9aaf43a69dc2cfaab5b306b8f91}, then all GL functions must be retrieved this way. Usually this is used to retrieve function pointers to Open\+GL extensions.

There are some quirks to looking up Open\+GL functions that require some extra care from the application. If you code carefully, you can handle these quirks without any platform-\/specific code, though\+:


\begin{DoxyItemize}
\item On Windows, function pointers are specific to the current GL context; this means you need to have created a GL context and made it current before calling \doxyref{S\+D\+L\+\_\+\+G\+L\+\_\+\+Get\+Proc\+Address()}{p.}{_s_d_l__video_8h_a6329bc32811639fe6f71606801cd9e7b}. If you recreate your context or create a second context, you should assume that any existing function pointers aren\textquotesingle{}t valid to use with it. This is (currently) a Windows-\/specific limitation, and in practice lots of drivers don\textquotesingle{}t suffer this limitation, but it is still the way the wgl A\+PI is documented to work and you should expect crashes if you don\textquotesingle{}t respect it. Store a copy of the function pointers that comes and goes with context lifespan.
\item On X11, function pointers returned by this function are valid for any context, and can even be looked up before a context is created at all. This means that, for at least some common Open\+GL implementations, if you look up a function that doesn\textquotesingle{}t exist, you\textquotesingle{}ll get a non-\/\+N\+U\+LL result that is {\itshape N\+OT} safe to call. You must always make sure the function is actually available for a given GL context before calling it, by checking for the existence of the appropriate extension with \doxyref{S\+D\+L\+\_\+\+G\+L\+\_\+\+Extension\+Supported()}{p.}{_s_d_l__video_8h_a1fa0cc176fbc16db139b40a121e9cfa7}, or verifying that the version of Open\+GL you\textquotesingle{}re using offers the function as core functionality.
\item Some Open\+GL drivers, on all platforms, {\itshape will} return N\+U\+LL if a function isn\textquotesingle{}t supported, but you can\textquotesingle{}t count on this behavior. Check for extensions you use, and if you get a N\+U\+LL anyway, act as if that extension wasn\textquotesingle{}t available. This is probably a bug in the driver, but you can code defensively for this scenario anyhow.
\item Just because you\textquotesingle{}re on Linux/\+Unix, don\textquotesingle{}t assume you\textquotesingle{}ll be using X11. Next-\/gen display servers are waiting to replace it, and may or may not make the same promises about function pointers.
\item Open\+GL function pointers must be declared {\ttfamily A\+P\+I\+E\+N\+T\+RY} as in the example code. This will ensure the proper calling convention is followed on platforms where this matters (Win32) thereby avoiding stack corruption.
\end{DoxyItemize}

Set the swap interval for the current Open\+GL context.

Some systems allow specifying -\/1 for the interval, to enable adaptive vsync. Adaptive vsync works the same as vsync, but if you\textquotesingle{}ve already missed the vertical retrace for a given frame, it swaps buffers immediately, which might be less jarring for the user during occasional framerate drops. If an application requests adaptive vsync and the system does not support it, this function will fail and return -\/1. In such a case, you should probably retry the call with 1 for the interval.

Adaptive vsync is implemented for some glX drivers with G\+L\+X\+\_\+\+E\+X\+T\+\_\+swap\+\_\+control\+\_\+tear\+:

{\footnotesize\ttfamily D\+E\+C\+L\+S\+P\+EC int S\+D\+L\+C\+A\+LL S\+D\+L\+\_\+\+Set\+Window\+Hit\+Test (\begin{DoxyParamCaption}\item[{\textbf{ S\+D\+L\+\_\+\+Window} $\ast$}]{window,  }\item[{\textbf{ S\+D\+L\+\_\+\+Hit\+Test}}]{callback,  }\item[{void $\ast$}]{callback\+\_\+data }\end{DoxyParamCaption})}

Provide a callback that decides if a window region has special properties.

Normally windows are dragged and resized by decorations provided by the system window manager (a title bar, borders, etc), but for some apps, it makes sense to drag them from somewhere else inside the window itself; for example, one might have a borderless window that wants to be draggable from any part, or simulate its own title bar, etc.

This function lets the app provide a callback that designates pieces of a given window as special. This callback is run during event processing if we need to tell the OS to treat a region of the window specially; the use of this callback is known as \char`\"{}hit testing.\char`\"{}

Mouse input may not be delivered to your application if it is within a special area; the OS will often apply that input to moving the window or resizing the window and not deliver it to the application.

Specifying N\+U\+LL for a callback disables hit-\/testing. Hit-\/testing is disabled by default.

Platforms that don\textquotesingle{}t support this functionality will return -\/1 unconditionally, even if you\textquotesingle{}re attempting to disable hit-\/testing.

Your callback may fire at any time, and its firing does not indicate any specific behavior (for example, on Windows, this certainly might fire when the OS is deciding whether to drag your window, but it fires for lots of other reasons, too, some unrelated to anything you probably care about {\itshape and when the mouse isn\textquotesingle{}t actually at the location it is testing}). Since this can fire at any time, you should try to keep your callback efficient, devoid of allocations, etc.
