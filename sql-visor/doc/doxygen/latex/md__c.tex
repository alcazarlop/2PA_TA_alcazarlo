{\bfseries{The backends/ folder contains backends for popular platforms/graphics A\+PI, which you can use in your application or engine to easily integrate Dear Im\+Gui.}} Each backend is typically self-\/contained in a pair of files\+: imgui\+\_\+impl\+\_\+\+X\+X\+X\+X.\+cpp + imgui\+\_\+impl\+\_\+\+X\+X\+X\+X.\+h.


\begin{DoxyItemize}
\item The \textquotesingle{}Platform\textquotesingle{} backends are in charge of\+: mouse/keyboard/gamepad inputs, cursor shape, timing, windowing.~\newline
 e.\+g. Windows ({\texttt{ imgui\+\_\+impl\+\_\+win32.\+cpp}}), G\+L\+FW ({\texttt{ imgui\+\_\+impl\+\_\+glfw.\+cpp}}), S\+D\+L2 ({\texttt{ imgui\+\_\+impl\+\_\+sdl.\+cpp}}), etc.
\item The \textquotesingle{}Renderer\textquotesingle{} backends are in charge of\+: creating atlas texture, rendering imgui draw data.~\newline
 e.\+g. Direct\+X11 ({\texttt{ imgui\+\_\+impl\+\_\+dx11.\+cpp}}), Open\+G\+L/\+Web\+GL ({\texttt{ imgui\+\_\+impl\+\_\+opengl3.\+cpp}}, Vulkan ({\texttt{ imgui\+\_\+impl\+\_\+vulkan.\+cpp}}, etc.
\item For some high-\/level frameworks, a single backend usually handle both \textquotesingle{}Platform\textquotesingle{} and \textquotesingle{}Renderer\textquotesingle{} parts.~\newline
 e.\+g. Allegro 5 ({\texttt{ imgui\+\_\+impl\+\_\+allegro5.\+cpp}}), Marmalade ({\texttt{ imgui\+\_\+impl\+\_\+marmalade.\+cpp}}). If you end up creating a custom backend for your engine, you may want to do the same.
\end{DoxyItemize}

An application usually combines 1 Platform backend + 1 Renderer backend + main Dear Im\+Gui sources. For example, the {\texttt{ example\+\_\+win32\+\_\+directx11}} application combines imgui\+\_\+impl\+\_\+win32.\+cpp + imgui\+\_\+impl\+\_\+dx11.\+cpp. There are 20+ examples in the {\texttt{ examples/}} folder. See {\texttt{ E\+X\+A\+M\+P\+L\+E\+S.\+MD}} for details.

Dear Im\+Gui is highly portable and only requires a few things to run and render, typically\+:


\begin{DoxyItemize}
\item Required\+: providing mouse/keyboard inputs (fed into the {\ttfamily \doxyref{Im\+Gui\+IO}{p.}{struct_im_gui_i_o}} structure).
\item Required\+: uploading the font atlas texture into graphics memory.
\item Required\+: rendering indexed textured triangles with a clipping rectangle.
\end{DoxyItemize}

Extra features are opt-\/in, our backends try to support as many as possible\+:


\begin{DoxyItemize}
\item Optional\+: custom texture binding support.
\item Optional\+: clipboard support.
\item Optional\+: gamepad support.
\item Optional\+: mouse cursor shape support.
\item Optional\+: I\+ME support.
\item Optional\+: multi-\/viewports support. etc.
\end{DoxyItemize}

This is essentially what each backends are doing + obligatory portability cruft. Using default backends ensure you can get all those features including the ones that would be harder to implement on your side (e.\+g. multi-\/viewports support).

It is important to understand the difference between the core Dear Im\+Gui library (files in the root folder) and backends which we are describing here (backends/ folder).


\begin{DoxyItemize}
\item Some issues may only be backend or platform specific.
\item You should be able to write backends for pretty much any platform and any 3D graphics A\+PI. e.\+g. you can get creative and use software rendering or render remotely on a different machine.

\doxysubsubsection{Using a custom engine?}\label{md__c_autotoc_md6}
You will likely be tempted to start by rewrite your own backend using your own custom/high-\/level facilities...~\newline
 Think twice!

If you are new to Dear Im\+Gui, first try using the existing backends as-\/is. You will save lots of time integrating the library. You can L\+A\+T\+ER decide to rewrite yourself a custom backend if you really need to. In most situations, custom backends have less features and more bugs than the standard backends we provide. If you want portability, you can use multiple backends and choose between them either at compile time or at runtime.

{\bfseries{Example A}}\+: your engine is built over Windows + Direct\+X11 but you have your own high-\/level rendering system layered over Direct\+X11.~\newline
 Suggestion\+: try using imgui\+\_\+impl\+\_\+win32.\+cpp + imgui\+\_\+impl\+\_\+dx11.\+cpp first. Once it works, if you really need it you can replace the imgui\+\_\+impl\+\_\+dx11.\+cpp code with a custom renderer using your own rendering functions, and keep using the standard Win32 code etc.

{\bfseries{Example B}}\+: your engine runs on Windows, Mac, Linux and uses Direct\+X11, Metal, Vulkan respectively.~\newline
 Suggestion\+: use multiple generic backends! Once it works, if you really need it you can replace parts of backends with your own abstractions.

{\bfseries{Example C}}\+: your engine runs on platforms we can\textquotesingle{}t provide public backends for (e.\+g. P\+S4/\+P\+S5, Switch), and you have high-\/level systems everywhere.~\newline
 Suggestion\+: try using a non-\/portable backend first (e.\+g. win32 + underlying graphics A\+PI) to get your desktop builds working first. This will get you running faster and get your acquainted with how Dear Im\+Gui works and is setup. You can then rewrite a custom backend using your own engine A\+PI.

Also\+: The {\texttt{ multi-\/viewports feature}} of the \textquotesingle{}docking\textquotesingle{} branch allows Dear Im\+Gui windows to be seamlessly detached from the main application window. This is achieved using an extra layer to the Platform and Renderer backends, which allows Dear Im\+Gui to communicate platform-\/specific requests such as\+: \char`\"{}create an additional O\+S window\char`\"{}, \char`\"{}create a render context\char`\"{}, \char`\"{}get the O\+S position of this
window\char`\"{} etc. See \textquotesingle{}Im\+Gui\+Platform\+IO\textquotesingle{} for details. Supporting the multi-\/viewports feature correctly using 100\% of your own abstractions is more difficult than supporting single-\/viewport. If you decide to use unmodified imgui\+\_\+impl\+\_\+\+X\+X\+X\+X.\+cpp files, you can automatically benefit from improvements and fixes related to viewports and platform windows without extra work on your side.

{\bfseries{About mouse cursor latency}}

Dear Im\+Gui has no particular extra lag for most behaviors, e.\+g. the value of \textquotesingle{}io.\+Mouse\+Pos\textquotesingle{} provided at the time of New\+Frame() will result in windows being moved to the right spot at the time of End\+Frame()/\+Render(). At 60 F\+PS your experience should be pleasant.

However, consider that OS mouse cursors are typically drawn through a very specific hardware accelerated path and will feel smoother than the majority of contents rendered via regular graphics A\+PI (including, but not limited to Dear Im\+Gui windows). Because UI rendering and interaction happens on the same plane as the mouse, that disconnect may be jarring to particularly sensitive users. You may experiment with enabling the io.\+Mouse\+Draw\+Cursor flag to request Dear Im\+Gui to draw a mouse cursor using the regular graphics A\+PI, to help you visualize the difference between a \char`\"{}hardware\char`\"{} cursor and a regularly rendered software cursor. However, rendering a mouse cursor at 60 F\+PS will feel sluggish so you likely won\textquotesingle{}t want to enable that at all times. It might be beneficial for the user experience to switch to a software rendered cursor {\itshape only} when an interactive drag is in progress.

Note that some setup or G\+PU drivers are likely to be causing extra display lag depending on their settings. If you feel that dragging windows feels laggy and you are not sure what the cause is\+: try to build a simple drawing a flat 2D shape directly under the mouse cursor!

{\bfseries{Font Atlas too large?}}


\begin{DoxyItemize}
\item If you have very large number of glyphs or multiple fonts, the texture may become too big for your graphics A\+PI. The typical result of failing to upload a texture is if every glyphs appears as white rectangles.
\item Mind the fact that some graphics drivers have texture size limitation. If you are building a PC application, mind the fact that your users may use hardware with lower limitations than yours.
\end{DoxyItemize}

Some solutions\+:


\begin{DoxyEnumerate}
\item Reduce glyphs ranges by calculating them from source localization data. You can use the {\ttfamily \doxyref{Im\+Font\+Glyph\+Ranges\+Builder}{p.}{struct_im_font_glyph_ranges_builder}} for this purpose and rebuilding your atlas between frames when new characters are needed. This will be the biggest win!
\item You may reduce oversampling, e.\+g. {\ttfamily font\+\_\+config.\+OversampleH = 2}, this will largely reduce your texture size. Note that while OversampleH = 2 looks visibly very close to 3 in most situations, with OversampleH = 1 the quality drop will be noticeable.
\item Set {\ttfamily io.\+Fonts.\+Tex\+Desired\+Width} to specify a texture width to minimize texture height (see comment in {\ttfamily Im\+Font\+Atlas\+::\+Build()} function).
\item Set {\ttfamily io.\+Fonts.\+Flags $\vert$= Im\+Font\+Atlas\+Flags\+\_\+\+No\+Power\+Of\+Two\+Height;} to disable rounding the texture height to the next power of two.
\item Read about oversampling {\texttt{ here}}.
\item To support the extended range of unicode beyond 0x\+F\+F\+FF (e.\+g. emoticons, dingbats, symbols, shapes, ancient languages, etc...) add {\ttfamily \#define I\+M\+G\+U\+I\+\_\+\+U\+S\+E\+\_\+\+W\+C\+H\+A\+R32}in your {\ttfamily \doxyref{imconfig.\+h}{p.}{imconfig_8h_source}}.
\end{DoxyEnumerate}

{\bfseries{Please note that many new C/\+C++ users have issues their files {\itshape because the filename they provide is wrong}.}}

Two things to watch for\+:
\begin{DoxyItemize}
\item Make sure your I\+D\+E/debugger settings starts your executable from the right working directory. In Visual Studio you can change your working directory in project {\ttfamily Properties $>$ General $>$ Debugging $>$ Working Directory}. People assume that their execution will start from the root folder of the project, where by default it oftens start from the folder where object or executable files are stored. 
\begin{DoxyCode}{0}
\DoxyCodeLine{\textcolor{comment}{// Relative filename depends on your Working Directory when running your program!}}
\DoxyCodeLine{io.Fonts-\/>AddFontFromFileTTF(\textcolor{stringliteral}{"MyImage01.jpg"}, ...);}
\DoxyCodeLine{}
\DoxyCodeLine{\textcolor{comment}{// Load from the parent folder of your Working Directory}}
\DoxyCodeLine{io.Fonts-\/>AddFontFromFileTTF(\textcolor{stringliteral}{"../MyImage01.jpg"}, ...);}
\end{DoxyCode}

\item In C/\+C++ and most programming languages if you want to use a backslash {\ttfamily \textbackslash{}} within a string literal, you need to write it double backslash {\ttfamily \textbackslash{}\textbackslash{}}. At it happens, Windows uses backslashes as a path separator, so be mindful. 
\begin{DoxyCode}{0}
\DoxyCodeLine{io.Fonts-\/>AddFontFromFileTTF(\textcolor{stringliteral}{"MyFiles\(\backslash\)MyImage01.jpg"}, ...);   \textcolor{comment}{// This is INCORRECT!!}}
\DoxyCodeLine{io.Fonts-\/>AddFontFromFileTTF(\textcolor{stringliteral}{"MyFiles\(\backslash\)\(\backslash\)MyImage01.jpg"}, ...);  \textcolor{comment}{// This is CORRECT}}
\end{DoxyCode}
 In some situations, you may also use {\ttfamily /} path separator under Windows.
\end{DoxyItemize}
